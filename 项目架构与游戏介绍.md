# 星梦之路项目架构与游戏介绍

## 一、项目定位

《星梦之路》是一款以娱乐圈成长为主题的单页网页文字游戏。玩家从新人阶段起步，在训练、跑通告、管理体力、经营口碑、推进角色关系、处理舆论风波之间不断做选择，逐步塑造自己的明星路线。

这不是单纯的数值堆叠游戏，也不是只有恋爱线的剧情游戏，而是把下面几类体验揉在一起：

- 明星养成：提升颜值、演技、唱功、舞蹈、魅力、人气、声望。
- 通告经营：接项目、看门槛、算收益、控节奏、冲阶段目标。
- 角色关系：通过邂逅、闲聊、特殊事件推进角色线和关系阶段。
- 舆论博弈：行业风向、热搜、口碑、黑稿、声明与反击共同影响节奏。
- 轻经营模拟：公司签约、团队招聘、服装搭配、设施升级、任务路线并行推进。

一句话概括：

《星梦之路》是一个“娱乐圈成长模拟 + 多角色关系推进 + 舆论与资源博弈 + 单页状态驱动叙事”的 React 游戏项目。

---

## 二、技术架构总览

### 1. 技术栈

- 前端框架：React 19
- 构建工具：Vite 6
- 状态管理：Zustand
- 样式：全局 CSS + 项目内自定义界面样式
- 动画与交互：以 React 组件交互和局部动态效果为主
- AI 能力：`src/services/` 下的 Gemini / OpenAI-compatible 服务封装
- 部署：Netlify

### 2. 架构形态

当前项目是典型的“单入口主控 + 多数据模块支撑”结构：

- 应用入口在 `src/main.tsx`
- 主游戏容器在 `src/App.tsx`
- 玩法配置集中在 `src/gameData.ts`
- 剧情与事件文本分散在 `src/stories/`
- 关系状态与事件触发逻辑集中在 `src/store/useGameStore.ts`
- 周聊天、员工、阶段任务等补充数据在 `src/data/`

这套结构的优点很明确：

- 迭代快，新增玩法时不需要先搭很重的基础设施
- 玩法逻辑与 UI 贴得近，调试体验直接
- 静态数据足够多，即使 AI 不可用，核心流程也能完整运行

当前架构的代价也同样明显：

- `src/App.tsx` 体量较大，承载了大量页面切换、游戏计算与 UI 拼装职责
- `src/gameData.ts` 既有类型、又有配置、又有剧情门槛，后期维护成本会逐步升高
- 一部分规则已经模块化，一部分仍然留在主文件中，边界还不够统一

## 三、目录结构与职责

### 根目录

- `package.json`
  依赖、脚本、构建入口。
- `vite.config.ts`
  Vite 构建配置与打包策略。
- `netlify.toml`
  Netlify 部署与缓存策略。
- `docs/项目架构与游戏介绍.md`
  本文档，面向项目说明与维护梳理。

### `src/`

- `main.tsx`
  React 挂载入口，挂载游戏视口与全局音乐播放器。
- `App.tsx`
  当前项目的核心主控文件，负责游戏主循环、页面切换、行动执行、通告处理、行业事件、日志、弹窗与大量 UI 渲染。
- `index.css`
  全局样式、主题基础、界面表现细节。

### `src/store/`

- `useGameStore.ts`
  角色关系状态、邂逅解锁、特殊事件触发、每周闲聊标记、剧情激活状态等跨模块共享数据的核心来源。
- `useBgmPlayer.ts`
  背景音乐播放器状态。

### `src/stories/`

- `encounters.ts`
  角色初遇与邂逅剧情。
- `daily.ts`
  日常事件池，按地点和通告触发。
- `dailyChats.ts`
  每周随机闲聊构造器，负责把聊天数据转换成剧情节点。
- `specialEvents.ts`
  角色专属关键事件与长剧情。
- `endings.ts`
  结局条件与分支判定。
- `announcements.ts`
  公告类剧情内容。
- `index.ts`
  剧情统一导出入口。

### `src/data/`

- `chatData.ts`
  每周闲聊主题、选项、回复、好感变动配置。
- `staffData.ts`
  员工池、员工工资、团队增益效果。
- `tasks.ts`
  阶段任务与推进目标。

### `src/gameData.ts`

这是当前项目最重要的配置中心，包含：

- 属性与时间类型
- 行动与地点定义
- 通告池
- 公司、背景、天赋、服装、设施
- 角色档案与图像
- 邂逅配置 `ENCOUNTER_CONFIG`
- 特殊事件配置 `SPECIAL_EVENT_CONFIG`
- 行业事件与部分规则常量

## 四、运行流程与状态流

### 1. 应用启动

应用从 `src/main.tsx` 启动，挂载：

- 游戏视口 `GameViewport`
- 主游戏组件 `App`
- 全局音乐播放器 `GlobalBgmPlayer`

### 2. 建档与开局

玩家在开局阶段完成：

- 名称与头像设定
- 出身背景选择
- 初始天赋选择
- 公司或路线选择

这些选择会在开局时直接影响：

- 初始金钱
- 初始属性
- 体力上限
- 人气和魅力起点

### 3. 主游戏循环

玩家进入正式流程后，核心循环基本是：

1. 选择地点
2. 执行动作或接取通告
3. 扣除体力与金钱
4. 获得属性、声望、人气或收益
5. 推进时间
6. 检查日常事件、邂逅、特殊事件、行业波动与任务进展
7. 写入日志并刷新界面

### 4. 页面切换方式

项目不是多路由页面，而是通过 `currentLocation` 和局部状态切换界面，属于游戏型 SPA 常见做法。

常见地点包括：

- `home`：我的小屋
- `company`：经纪公司
- `city`：城市活动
- `jobs`：通告大厅
- `schedule`：行程安排
- `wardrobe`：衣橱
- `trophy`：成就/奖杯
- `quests`：任务
- `social`：社交平台
- `contacts`：联系人/私聊

## 五、核心数据模型

### 1. 玩家主状态

`src/gameData.ts` 中的 `Player` 是运行时核心数据结构之一，主要包含：

- 基础资源：`stamina`、`maxStamina`、`money`
- 主属性：`appearance`、`acting`、`singing`、`dancing`、`charm`、`popularity`、`mood`
- 扩展指标：`reputation`
- 社交数据：粉丝、发帖、会话、关系值
- 装备与设施：服装、设施等级
- 进度数据：已完成任务、已接通告、已完成通告、荣誉等

### 2. 角色关系状态

`src/store/useGameStore.ts` 中的 `CharacterPsychology` 是角色线的核心模型，包含：

- `favor`：好感度
- `paranoia`：猜疑
- `possessiveness`：占有欲
- `insecurity`：不安
- `trust`：信任
- `keyEventTriggered`：是否已解锁关键事件后的上限
- `favorLocked`：好感是否被锁在关键事件前上限
- `stage`：心理阶段
- `completedEvents`：已完成角色事件

这说明本项目的关系系统不是单一好感条，而是“好感主导 + 心理侧写辅助”的结构。

### 3. 关系阶段映射

内部心理阶段：

- `guarded`
- `warming`
- `dependent`
- `obsessive`
- `fractured`

展示层关系阶段：

- `guarded` -> `陌生`
- `warming` -> `熟悉`
- `dependent` -> `暧昧`
- `obsessive` -> `心动`
- `fractured` -> `陌生`

也就是说，表面看到的是“陌生/熟悉/暧昧/心动”，底层实际上还保留了角色心理结构和异常态。

## 六、玩法主系统说明

### 1. 属性成长系统

玩家通过行动、训练、通告、奖励提升属性。主要成长方向包括：

- 颜值：影响镜头感、时尚与部分商业项目
- 演技：影响影视类通告与角色戏份
- 唱功：影响音乐与舞台类项目
- 舞蹈：影响偶像舞台、综艺舞台、竞技类表演
- 魅力：影响商业、社交、品牌表达
- 人气：决定热度层级与高阶项目准入
- 声望：决定职业稳定度、公众观感与部分行业评价

项目里很多地点还有成长增益，例如：

- 居酒屋、旧院走廊偏向好感成长
- 录音棚、电台室、电竞包厢、公益影像展偏向声望或人气成长

也就是说，地点不只是背景板，而是成长倍率系统的一部分。

### 2. 通告与工作系统

通告定义集中在 `src/gameData.ts` 的 `JOBS` 数据中，每个通告至少包含：

- 名称与类型
- 角色定位/项目定位
- 属性门槛
- 体力消耗
- 时间消耗
- 金钱与人气奖励
- 额外属性奖励
- 类别和标签

玩家接通告时会检查：

- 属性是否达标
- 体力是否足够
- 金钱是否足够
- 某些特殊通告是否关联角色邂逅

如果条件不满足，系统不只是报错，还会追加“系统嘲讽”日志，强化娱乐圈压力感。

### 3. 日程系统

玩家可以把通告加入日程，之后统一执行。日程系统承担两类体验：

- 节奏管理：提前排布资源
- 风险感：如果硬接太多项目，容易体力不足、收益不稳、被系统嘲讽

### 4. 公司与团队系统

公司系统负责长期经营向的差异化。

公司配置包括：

- 抽成比例
- 初始加成
- 最低门槛
- 合约时长
- 解约惩罚
- 公司路线特色

员工系统在 `src/data/staffData.ts`，例如：

- PR Manager：抵消一次负面声望下滑
- Stylist：降低颜值/魅力门槛
- Scout：有概率刷出隐藏高阶通告
- Legal Counsel：减轻解约与商业损失

这套系统让中后期玩法不只是刷属性，也要考虑经营效率。

### 5. 衣橱与搭配系统

服装不仅是收集内容，也会提供：

- 属性加成
- 风格标签
- 与通告标签匹配时的额外加成

这意味着衣橱系统有明确功能性，不是纯展示。

## 七、角色系统与好感度判定

### 1. 好感度不是唯一判定

项目当前的角色关系推进以 `favor` 为主，但不是只看好感数值。底层阶段判定依赖五个心理指标：

- favor
- paranoia
- possessiveness
- insecurity
- trust

阶段计算规则写在 `useGameStore.ts` 的 `deriveStage` 中，大致逻辑如下：

- 猜疑或占有欲过高，且好感足够高时，会进入偏执型阶段 `obsessive`
- 不安很高且信任很低时，会进入破裂态 `fractured`
- 好感高、信任高、不安较低时，会进入稳定亲密阶段 `dependent`
- 好感或信任达到一定值时，会从戒备转为升温 `warming`

所以项目的好感系统本质上是：

“数值增长”决定能否推进，
“心理结构”决定推进后的关系气质。

### 2. 好感上限锁

当前实现里存在一个非常关键的机制：

- 在关键事件触发前，角色好感上限默认为 80
- 只有触发 `keyEventTriggered` 后，角色好感上限才开放到 100

这意味着：

- 玩家不能靠无限刷日常把角色线硬推满
- 必须进入关键剧情节点，关系才会正式突破

这也是本项目避免“纯刷数值恋爱”的重要设计。

### 3. 特殊事件判定

特殊事件由 `SPECIAL_EVENT_CONFIG` 定义，判定条件包含：

- 角色 ID
- 最低好感 `favorMin`
- 最高好感 `favorMax`（部分可选）
- 前置事件 `requiredPrevEvent`
- 关系阶段 `stageRequired`

当前配置已经给出了部分明确门槛，例如：

- 沈默事件 1：好感至少 20
- 沈默事件 2：好感至少 40，且已完成前一事件
- 柳梦瑶事件链：30 -> 50 -> 70 -> 85 逐段推进

这套配置说明角色线是严格分段推进的，不是随机跳剧情。

### 4. 阶段差分反馈

很多剧情选项都不是固定回复，而是根据阶段返回不同文本。

例如：

- `StoryOption.stageFollowUp`
- `specialEvents.ts` 中按 `陌生/熟悉/暧昧/心动` 给不同结果
- `dailyChats.ts` 会为同一句回复自动包装不同阶段前缀

也就是说，同一个选择在不同关系阶段下，读感和风险都不同。

这正是“好感度判定”真正生效的地方：

- 不只是数值加减
- 而是影响剧情语气、角色边界、回应强度和事件质感

## 八、日常事件系统

### 1. 日常事件来源

日常事件主要在 `src/stories/daily.ts` 中定义。

事件由 `DailyEventConfig` 描述，包含：

- 事件 ID
- 角色 ID
- 标题
- 场景文本
- 台词
- 选项
- 触发器
- 触发概率

### 2. 触发方式

日常事件有两类入口：

- 按通告触发：`getDailyEventsByJob(jobId)`
- 按地点触发：`getDailyEventsByLocation(locationId)`

也就是说，玩家做什么事、去了哪里，会直接决定更容易触发谁的日常事件。

### 3. 触发规则

日常事件并不是必出，而是：

- 先筛选符合地点或通告的事件池
- 再按 `triggerChance` 概率判定
- 最后只随机抽取一个

因此日常事件的体验目标不是“稳定刷分”，而是“在主循环里制造关系波纹”。

### 4. 日常事件的作用

日常事件通常承担三类功能：

- 轻度补好感
- 补足角色日常相处感
- 为关键事件前的情绪积累提供缓冲层

柳梦瑶这类角色还写了更细的专属日常包，说明角色表达强度可以按人物单独定制。

## 九、每周闲聊与联系人系统

### 1. 每周闲聊限制

每个角色每周只能聊一次，状态由 `useGameStore.ts` 中的 `hasChattedThisWeek` 记录。

对应操作包括：

- `markChattedThisWeek`
- `resetWeeklyChatFlags`

这让聊天系统具备了节奏控制，不会变成无限刷好感按钮。

### 2. 闲聊数据结构

闲聊数据在 `src/data/chatData.ts` 中维护，每条聊天主题包含：

- 场景
- 开场白
- 三个选项
- 角色回复
- 好感变化值 `affection`

### 3. 闲聊系统功能定位

每周闲聊不是长剧情，而是“短反馈 + 角色性格校准器”。

它主要负责：

- 给玩家稳定关系反馈
- 强化角色口吻
- 用轻量选择测试玩家对角色的理解
- 提供持续的好感增减

### 4. 日常闲聊和特殊剧情的区别

- 闲聊：短、频繁、每周一次、偏日常口吻
- 日常事件：绑定地点或工作，随机触发，偏相处场景
- 特殊事件：严格门槛、强剧情、角色线核心推进

三者组合后，角色体验会显得更完整。

## 十、邂逅系统

### 1. 邂逅的设计目标

邂逅是角色线入口，不是简单“解锁通讯录”。

它负责：

- 让每个角色有明确登场舞台
- 把角色与职业路线、时间窗口绑定
- 形成“错过就可能真的错过”的娱乐圈感

### 2. 判定字段

`ENCOUNTER_CONFIG` 的核心条件包括：

- 出现地点
- 年份和周数范围
- 指定通告前置
- 人气要求
- 前置角色解锁
- 前置角色好感要求
- 是否可错过 `isMissable`

### 3. 错过机制

项目里已经实现错过判定：

- 如果邂逅事件带有 `isMissable`
- 且时间窗结束时仍未触发
- 该角色状态会从 `locked` 变成 `missed`

这使得角色解锁更像真实路线选择，而不是迟早全收集。

## 十一、舆论、行业事件与嘲讽系统

### 1. 行业事件系统

`App.tsx` 中存在 `industryEvent` 状态，用来承载行业风向或舆论波动弹窗。

行业事件结构包含：

- 标题
- 场景说明
- 波动等级
- 风向情绪
- 事件类型
- 来源标签
- 多个可选策略

每个选择都会带来不同奖励或代价，例如：

- 声望变化
- 人气变化
- 属性变化
- 金钱变化

这套系统的作用是把娱乐圈中的“环境压力”从背景描述变成玩家要处理的显式决策。

### 2. 热搜与公众观感

项目多个系统文本都直接围绕热搜、黑稿、声明、翻盘来写，体现在：

- 周聊天主题
- 日常事件文本
- 通告描述
- 行业事件场景
- 阶段任务文案

也就是说，“热度”和“口碑”不是一个数值的两种叫法，而是玩法气氛层面反复施压的世界规则。

### 3. 系统嘲讽机制

`App.tsx` 中已经明确实现 `taunt` 类型日志。

典型触发场景包括：

- 体力不足却强行动作
- 金钱不足却硬接项目
- 属性不足却尝试高门槛通告
- 日程执行失败

表现形式通常是：

- 先写一条失败日志
- 再追加一条 `【系统嘲讽】...`

这套机制的价值不是单纯搞笑，而是强化项目的情绪质地：

- 娱乐圈不只奖励你，也会嘲笑你
- 失败不是静默的，而是有社会压力感的
- 玩家会更清楚“盲冲”和“计划”之间的差别

如果后续继续扩展，这个系统非常适合再接：

- 不同阶段的专属嘲讽文本
- 按角色路线区分的失败语气
- 舆论事件失败后的群体反馈

### 4. 日常事件与嘲讽的关系

项目很有意思的一点在于：

- 日常事件负责提供情绪缓冲和关系补偿
- 嘲讽系统负责提供压力和失败羞耻

两者配合后，整体节奏会更像“高压圈层里艰难成长”，而不是平滑升级。

---

## 十二、结局系统

结局定义在 `src/stories/endings.ts`。

当前结局判定不是只看单一好感，而是组合判断：

- 角色好感是否足够高
- 特殊事件是否完成
- 关键选项是否选对
- 属性或职业条件是否满足
- 某些布尔标记是否被触发

这意味着结局不是简单的“刷满好感拿 HE”，而是：

- 数值
- 剧情选择
- 关键节点
- 角色路线理解

共同决定最终归宿。

---

## 十三、当前项目的架构优点

- 数据驱动程度已经不低，大部分核心玩法都能通过配置扩展
- 角色线层次清楚，邂逅、闲聊、日常、特殊事件、结局已经形成完整链条
- 游戏体验不只靠数值，还靠日志、舆论、口吻、事件压力构成情绪反馈
- AI 被设计成增强层而不是硬依赖，主流程稳定性更高
- 状态管理已经把角色线关键逻辑从 `App.tsx` 中抽离一部分，为后续重构留了基础

---

## 十四、当前项目的主要结构问题

### 1. `App.tsx` 过重

当前主文件同时承担了：

- 页面路由切换
- 行动结算
- 通告接取
- 行业事件处理
- 聊天处理
- 剧情弹窗处理
- 日志和大量 UI 渲染

后续继续加系统时，维护成本会明显上升。

### 2. `gameData.ts` 过大

现在它既是类型文件，又是配置中心，还是部分规则入口。中后期建议拆分为：

- `gameData/core.ts`
- `gameData/jobs.ts`
- `gameData/characters.ts`
- `gameData/encounters.ts`
- `gameData/specialEvents.ts`
- `gameData/items.ts`

### 3. 规则分散

目前规则分散在：

- `App.tsx`
- `useGameStore.ts`
- `gameData.ts`
- `stories/*`
- `data/*`

虽然还能维护，但已经接近需要建立更清晰“页面层 / 服务层 / 规则层 / 数据层”的阶段。

---

## 十五、建议中的下一步演进方向

### 短期建议

- 把 `App.tsx` 里的通告逻辑、行动逻辑、行业事件逻辑拆成 hooks 或 service
- 给“日常事件 / 闲聊 / 特殊事件 / 邂逅”建立统一触发说明和调试入口
- 把嘲讽文本、失败反馈、舆论反馈从主文件抽成独立文本库

### 中期建议

- 拆分 `gameData.ts`
- 建立统一的“关系效果”类型，未来让 paranoia / trust 等指标真正进入剧情分支
- 给行业事件建立独立数据池，而不是主要依赖主文件内流程处理

### 长期建议

- 把主循环结算抽成纯函数，降低 UI 和规则耦合
- 为角色线建立更标准的数据协议，减少超长手写剧情维护压力
- 增加自动化测试，尤其覆盖邂逅错过、好感锁、事件链门槛、结局判定

---

## 十六、总结

《星梦之路》当前已经不是“一个有页面的原型”，而是一个具备完整成长主循环的娱乐圈养成项目：

- 有明确的技术架构
- 有稳定的状态驱动模型
- 有角色线推进结构
- 有通告、经营、衣橱、员工、任务、结局等外围系统
- 有舆论、热搜、行业波动、系统嘲讽这些强化情绪体验的机制

尤其值得保留和继续放大的，是下面几条设计方向：

- 好感度不是孤立数字，而是带心理阶段的关系系统
- 日常事件不是摆设，而是角色线的节奏层
- 嘲讽系统让失败具备娱乐圈语境下的羞耻感和压迫感
- 舆论与行业风向把“成长”从单纯升级变成了公共舞台上的生存

如果后续继续迭代，建议把现有内容从“快速堆叠完成”进一步整理成“规则可复用、数据可扩展、结构可维护”的版本，这个项目会非常适合继续做大。
